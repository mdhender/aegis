        paging facility does not seem to work properly within xterms.
        If $LINES is set, it works fine, but using the TIOGWINSZ ioctl

Project "aegis.2.3", Change 20                                   Page 1
Change Details                                 Tue Mar 14 06:50:36 1995

NAME
        Project "aegis.2.3", Delta 26, Change 20.

SUMMARY
        locking "deadlock" possible

DESCRIPTION
         Christian F. Goetze <c-goetze@u-aizu.ac.jp> writes: "I
        recently had an interesting situation. I was automatically
        integrating a change in one window, and I tried to pass a
        review in another window. Both operations happened to be on
        the same change (because I forgot the -p option in the
        "aerpass" command). I watched how the message "waiting for
        change lock" got sent back and forth between the two
        windows... "

        I designed the locking to (theoretically) avoid problems like
        this. The dining philosophers problem (project lock, change
        lock, user lock, build lock, multiple processes wanting
        combinations of 2 or more) is solved by using a single
        "permission to lock" lock.  But while typing this I realize
        there *is* a race, which needs to be solved by a random
        backoff, rather than the minimum delay currently implemented.

        The race is: once the master lock is obtained, the process
        then attempts to grab all the locks it wants.  If it can get
        them, it releases the master lock.  If it can't get them all,
        it gives all but one back (the change lock, in your example)
        *and* releases the master lock and waits on the change lock
        (rather than waking up once a second and trying again).  Once
        the change lock becomes available, it is *released* and the
        cycle starts again.  Random backoff (a small number of
        seconds) would fix it, I think, i.e. combine the current
        strategic waiting and polling.

        This change also places project attribute suggestions in the
        notification shell scripts, and fixes a bug which provented
        the review-fail email script from working.

        This change also modifies the heuristic for detecting symbolic
        link loops - the old one kept getting false hits.

        This change is exempt from testing against the baseline.

ARCHITECTURES
        This change must build and test in each of the "alpha", "sun4"
        and "sun5" architectures.
